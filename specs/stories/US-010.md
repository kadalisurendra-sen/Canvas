# US-010: Template Wizard Step 3 — Fields & Sections

**As an** admin
**I want to** build a hierarchical field structure with sections and fields for each stage
**So that** evaluators have a structured form to fill out during assessments

## Acceptance Criteria

- [ ] AC1: Step 3 shows a left panel listing all stages (from Step 2) as tabs; clicking a stage shows its sections and fields
- [ ] AC2: Each stage has an "Add Section" button; sections display as collapsible groups with a name (editable inline) and sort order
- [ ] AC3: Each section has an "Add Field" button; fields display within their section with: field key, label, field type dropdown, help text, mandatory toggle, scoring toggle, sort order
- [ ] AC4: Field type dropdown includes: text_short, text_long, single_select, multi_select, number, date
- [ ] AC5: When field type is single_select or multi_select and scoring is enabled, an "Options" sub-panel appears allowing add/edit/remove of options with: label, value, score (numeric)
- [ ] AC6: A live preview panel on the right side renders how the fields will look to an evaluator
- [ ] AC7: `PUT /api/v1/templates/{template_id}/fields` accepts the full sections + fields structure and persists it (template_sections, template_fields, field_options tables)
- [ ] AC8: "Next" saves and advances to Step 4; "Back" returns to Step 2
- [ ] AC9: Validation: each section must have a name, each field must have a key (unique within template), label, and type

## Technical Notes
- This is the most complex wizard step; consider splitting the UI into sub-components (StageTabList, SectionEditor, FieldEditor, OptionsEditor, PreviewPanel)
- Backend receives a nested JSON payload: `{ stages: [{ stage_id, sections: [{ name, sort_order, fields: [{ field_key, label, field_type, ... options: [] }] }] }] }`
- Backend replaces all sections/fields/options in a single transaction
- Field key auto-generated from label (snake_case) but editable

## Dependencies
- US-009 (Wizard Step 2 — stages must exist)
